## Entity: User

Represents a registered person. The User entity is the sole business entity owned
by this service. All other services that need user identity rely on the `userId` encoded
in the JWT token — they do NOT query this service's database directly.

### Fields

| Field | Type | Constraints | Notes |
|-------|------|-------------|-------|
| `_id` | ObjectId | Auto-generated by MongoDB | Exposed as `id` (string) in API responses |
| `name` | String | Required; 1–100 characters | Full display name |
| `email` | String | Required; unique; valid email format | Primary human-readable identifier; lowercase-normalised before storage |
| `passwordHash` | String | Required | bcrypt hash (cost 12). **NEVER returned in any API response or log.** |
| `createdAt` | Date | Auto-set on insert | ISO 8601 timestamp |
| `updatedAt` | Date | Auto-updated on every save | ISO 8601 timestamp |

### Validation Rules

- `name`: non-empty string, max 100 characters, trimmed of leading/trailing whitespace.
- `email`: must pass RFC 5322 email format check; normalised to lowercase before storage and uniqueness check.
- `passwordHash`: populated by the service layer after bcrypt hashing — never set directly from request input.
- Minimum password (pre-hash): 8 characters. Enforced at the zod validation layer, not the model layer.

### State Transitions

Users have no explicit status field in this version. The only lifecycle events are:

```
[Not Exists] ──── POST /api/users ────► [Active]
[Active]     ──── PATCH /api/users/:id ► [Active] (fields updated)
[Active]     ──── DELETE /api/users/:id ► [Not Exists] (hard delete)
```

Soft-delete is out of scope (documented in spec Assumptions).

### Indexes

| Index | Fields | Type | Purpose |
|-------|--------|------|---------|
| Primary | `_id` | Unique (default) | Document identity |
| Email uniqueness | `email` | Unique | Enforce one account per email; supports login lookup |

### Mongoose Schema (TypeScript representation)

```typescript
// This is the authoritative shape — implementation must match this exactly.
interface IUser {
  _id: Types.ObjectId;
  name: string;
  email: string;          // stored lowercase
  passwordHash: string;   // NEVER serialised to API response
  createdAt: Date;
  updatedAt: Date;
}
```

The Mongoose schema MUST include a `toJSON` transform that:
1. Renames `_id` → `id` (string).
2. Removes `passwordHash` from the output object.
3. Removes `__v` (Mongoose version key).

### Public API Representation (safe user object)

What API responses return — no sensitive fields:

```json
{
  "id": "664f1a2b3c4d5e6f7a8b9c0d",
  "name": "Jane Doe",
  "email": "jane@example.com",
  "createdAt": "2026-05-26T10:00:00.000Z",
  "updatedAt": "2026-05-26T10:00:00.000Z"
}
```

## Redis Cache Schema

Redis is used as a read-through cache for user profile lookups. It is NOT the source
of truth — MongoDB is. Redis holds short-lived copies of the public user object only.

| Key Pattern | Value | TTL | Set On | Invalidated On |
|-------------|-------|-----|--------|----------------|
| `user:{userId}` | JSON string of public user object (see above) | `CACHE_TTL_SECONDS` (default: 300 s) | Successful `GET /api/users/:id` cache miss | `PATCH /api/users/:id` success, `DELETE /api/users/:id` success |

**Cache value format** (JSON-serialised string stored in Redis):

```json
{
  "id": "664f1a2b3c4d5e6f7a8b9c0d",
  "name": "Jane Doe",
  "email": "jane@example.com",
  "createdAt": "2026-05-26T10:00:00.000Z",
  "updatedAt": "2026-05-26T10:00:00.000Z"
}
```

**Important**: `passwordHash` MUST NOT appear in the cached value. The cache layer
serialises only the output of the Mongoose `toJSON` transform.

## Environment Variables

All connection strings and secrets MUST be sourced from environment variables.
No defaults are hard-coded for production-sensitive values.

| Variable | Required | Default (dev) | Description |
|----------|----------|---------------|-------------|
| `PORT` | No | `3001` | HTTP port the service listens on |
| `MONGODB_URI` | Yes | — | Full MongoDB connection string, e.g. `mongodb://localhost:27017/user-service-db` |
| `JWT_SECRET` | Yes | — | Shared HS256 signing secret (≥ 256 bits). Must match the Auth Service's secret. |
| `REDIS_URL` | Yes | — | Redis connection URL, e.g. `redis://localhost:6379` |
| `CACHE_TTL_SECONDS` | No | `300` | Redis key TTL for user profile cache entries |
| `BCRYPT_COST` | No | `12` | bcrypt cost factor (do not lower below 10 in production) |
| `NODE_ENV` | No | `development` | `development` / `production` / `test` |
| `LOG_LEVEL` | No | `info` | pino log level (`debug`, `info`, `warn`, `error`) |
